# ClaudeCoach - Methodology Audit

Read-only review of the automated training planner. Audit date 2026-07-02. All code paths relative to `/Users/diamondpeakconsulting/diamondpeak-site/ClaudeCoach`.

Athlete roles (names withheld per brief): **Athlete A** = long-course triathlon, race 2026-09-19, ~11 weeks out. **Athlete B** = mid-level half-distance triathlon, race 2026-09-20, cycle-tracked. **Athlete C** = beginner, one-day alpine sportive, 2026-08-29. Where a citation would name an athlete's directory it is written `athletes/<A>/`, `athletes/<B>/`, `athletes/<C>/`.

**Data caveat, read first.** Delivered-product reconstruction is bounded by the local logs: `athletes/*/session-log.json` is frozen at 2026-06-17 for A and B, and at 2026-05-21 for C (only 6 stub entries, none with real TSS). The prescription pipeline (`current-state.md`) is current to 2026-06-30, so the session-log ingestion path is roughly two weeks behind. Everything below therefore judges the system on intent plus partial delivery to mid-June, not full delivery to race-minus-11-weeks. Live CTL/ATL/TSB (from `training-data*.json`, generated 2026-07-02) are exact.

---

## 1. Executive verdict

The methodology is well-designed and, for Athlete A, well-tuned; the problem is not what the system decides but whether its decisions bind. Almost every deterministic safeguard the design relies on is present in code yet runs advisory: the daily readiness gate is LLM-mediated with the Python backstop in shadow mode (`scripts/daily-prescription.py:365-406`, `PRESCRIPTION_BACKSTOP=shadow`); the weekly ramp/TSS backstop is inert in the push path because `ctl_today` is never passed (`lib/plan_builder.py:106-108` versus `ironman-analysis/primitives/validate_plan.py:263`); the intensity, run-volume and ankle rules are structured only for Athlete A and degrade silently to permissive defaults for B and C. On top of that seam sit two things that are not encoded at all and must be built: programmed recovery/deload weeks (the plan is a monotonic CTL-chase, `lib/plan_tools.py:288-299`) and a shaped taper (the final four weeks have no TSS target and the taper intensity split reverts to base, `lib/session_library.py:238`). The single highest-value change is to make the guardrails bind: flip the deterministic layer from shadow to authoritative, pass the missing inputs so the existing checks fire at plan time, and bring the B and C athlete configs up to A's standard. That is largely an arming exercise. The one large build that arming cannot substitute for is recovery-week structure, which is the top training-soundness gap.

## 3. Prioritised findings

### P0 - outcome-critical

**P0-1. No programmed recovery or deload weeks; the plan is a monotonic CTL-chase.**
`lib/plan_tools.py:288-299` computes one smooth weekly target, `recommended_weekly_tss = min(required, safe)`, and nothing ever inserts a down-week. The documented "3 weeks load, 1 week recovery at 60-65% of the prior week" and "missed >30% -> next week is recovery" (`blueprints/blueprint.md:60-62`) exist nowhere in the planning path (grep for `recovery|deload|3:1` in `lib/plan_tools.py` and `lib/session_library.py` returns only docstrings). Athlete A's own projected trajectory climbs ~+3.6 CTL/week with zero dips. Physiological rationale: absence of planned unloading is the classic accumulation/overuse pattern, and it is most dangerous here because A has a confirmed partial ATFL+CFL ankle tear (`athletes/<A>/current-state.json`, ultrasound 2026-06-12). This is a build, not an arming. Recommendation: give `required_tss` a deload branch (every 3rd or 4th week at 60-65% of the preceding week, or triggered by a >30% miss) so the forward plan, not just the reactive watchdog, unloads.

**P0-2. The taper is unshaped in code during the four weeks that matter most.**
Once the training week passes `peak_end_week`, `required_tss` returns a taper note with no target (`lib/plan_tools.py:269-271`), so `weekly_tss_target` becomes `None`. Downstream every load check disengages: `close_to_target` returns the proposal unscaled (`scripts/stage1-plan.py:107`), the TSS audit is skipped and `load_on_target` is forced true (`scripts/stage1-plan.py:137,300`). The evidence-based step-down (blueprint `blueprints/blueprint.md:181-183`, "70% -> 55% -> 40% of peak, maintain intensity") is documented but never computed, and the taper intensity split reverts to base because events carry no taper TID row (`lib/session_library.py:238` falls back to `base`, e.g. 85/10/5 mostly-easy). The `volume_factor: 0.6` in `config/session-library.json` is read by nothing. Physiological rationale: taper meta-analyses (Bosquet 2007; Mujika) are unambiguous that an effective taper reduces volume while maintaining or raising intensity and frequency; reverting taper intensity to the base high-easy split is the opposite of consensus, and leaving volume entirely to LLM discretion in the final weeks is where a season is won or lost. Frame: the defect is "unmanaged in code", which is bulletproof; whether A's 4-week taper length is also too long is a secondary, more arguable point (see P1-5). Recommendation: add a taper branch to `required_tss` returning stepped weekly targets, and add a `taper` TID row that holds intensity.

**P0-3. The injury-safety machinery is plumbed for Athlete A only and degrades silently for B and C.**
The R1 ankle rule (`ironman-analysis/primitives/modulation.py:154-180`) reads pain and cleared-status from `current-state.json`'s `ankle` block via `scripts/daily-prescription.py:218-230`, which defaults to `(0, True)` (no pain, cleared) when the block is absent. Verified: `athletes/<B>/current-state.json` contains only `last_updated` and `menstrual_cycle`, no `ankle` key, and `athletes/<B>/profile.json` has `injuries: []`. Yet `athletes/<B>/current-state.md` narrates an active ankle block, and the 2026-07-02 prescription log records "run_quality BLOCKED (R1) ... Pushed blocked note event 120035084" followed by "ran Thu 2 Jul tempo anyway ... running cost +10.3% in final third". So the deterministic R1 could not have fired for B (no structured input); the block was LLM-narrated, the athlete overrode it, and the cost was only caught post-hoc. Provenance caveat: because the session logs are stale, the tempo run is corroborated only by the LLM-written state file, and B's injury status is contradictory across sources; treat the episode as illustration, not a proven injury event. The clean, code-verifiable defect is that the same safety rule protecting A is inert for B and C. Recommendation: move ankle/niggle state into structured per-athlete config for every athlete, or fail closed (treat missing state as "not cleared") rather than open.

**P0-4. The guardrails do not bind: the deterministic layer runs shadow-only or inert.**
This is the through-line. The daily gate that produces the athlete's session is LLM-mediated: the model reads the readiness prefetch, calls `modulate.py`, and pushes the result, while the server-side deterministic engine that computes the same prescription is non-authoritative by design (`scripts/daily-prescription.py:158-159,365-406`, "the LLM's own prescription is what reached the athlete"). One skipped or misclassified tool call means nothing gates. In parallel the weekly ramp/TSS backstop in `validate_week` is armed by no caller in the push path: `lib/plan_builder.py:106-108` passes `ramp_cap` but not `ctl_today`, and `weekly_tss_cap` is passed by no one, so both hard checks silently no-op (`ironman-analysis/primitives/validate_plan.py:251,263`). The docstring claims this validator exists "so a breach cannot reach the athlete just because the model graded its own homework" (`validate_plan.py:5-8`); that guarantee is currently void. Recommendation: flip `PRESCRIPTION_BACKSTOP` to authoritative (or block on divergence), and pass `ctl_today` and `weekly_tss_cap` into the `plan_builder` validate call. Both are small changes with large reliability leverage.

### P1 - meaningful fitness or safety left on the table

**P1-1. No execution-side intensity check; realised distribution drifts hard for B.** The distribution validator classifies sessions by name on the plan, not by realised zone (`ironman-analysis/primitives/validate_plan.py:137-168`, using `classify_session_type` in `modulation.py:106-143`), fires soft-only, on excess planned quality only, with a generous 12-point tolerance. Computed from the available logs (session-level effective IF, directional for runs): Athlete B ran 36% low / 51% moderate / 14% hard by time against an 78-83% low target, with 0 of 9 logged runs classifying easy, including sessions named "Easy Z2 run" at effective IF 0.75-0.78 and a "Z2" long ride at 0.83. Athlete A sat at 94% low / 6% moderate / 0% hard, more polarised than planned but with zero bike quality entering the Specific phase for a race whose own stated limiter is a bike fade (target bike IF 0.636 -> 0.71). The system validates the label, not the reality. Recommendation: add a realised-TID audit off actual activities (the data already flows through `activity-watcher`/`icu_fetch`), checking both excess quality and insufficient quality.

**P1-2. B's load spike is confounded by a probable threshold miscalibration (links dimensions 11, 1, 2).** B's last full logged week jumped 507 -> 755 TSS (+49%). Before reading that as a clean ramp breach, note dimension 11: her configured LTHR (191) may be inflating hrTSS, which would inflate both the weekly total and the moderate-zone share in P1-1. So the grey-zone creep, the load spike, and the stale-threshold finding are one interacting problem, not three independent ones. Recommendation: validate LTHR against recent data first, then re-check the ramp and distribution numbers.

**P1-3. The CTL-ramp cap is a weak acute-load guard, and no monotony/ACWR check exists.** Because CTL is 42-day smoothed, "within +4 CTL/week" for A permits ~800 TSS in a single week (above his own documented peak ceiling of 778); at the +7 overreach threshold, ~938. The blueprint's `max_weekly_tss = hours x 100 x IF^2` "hard upper bound" (`blueprints/blueprint.md`) exists only in `generate-blueprint.py` for display and is never passed to `validate_week`. There is no acute-to-chronic, Foster monotony, or SD-strain check in the planner (the ATL-gap and ramp flags in `ironman-analysis/primitives/load.py:384-449` are post-hoc watchdog-side). Recommendation: add a within-week acute-load or monotony guard, and pass the hours ceiling as `weekly_tss_cap`.

**P1-4. The race predictor is unreproducible from source and misattributes causation.** The served `training-data.json` carries a `racePredictor` anchored to bike IF 0.73, but `athletes/<A>/profile.json` has no `race_predictor` block at all, so a clean regeneration would omit it (`scripts/refresh-site-data.py:459,470-472,607-608`); the anchor is internally inconsistent (reports 201 W at IF 0.73, but 201/FTP 316 = 0.636, which is the profile's real `prev_race.bike_if`); and the private and public `training-data.json` copies have diverged (2026-05-26 versus 2026-07-02) despite being written in one run. This points to repo-versus-VM drift. Separately, the run split is modelled as `rmin = 42.2 * (thr / IF) / 60` (`refresh-site-data.py:485`), scaling the marathon time off the bike-derived IF, which contradicts the athlete's own race notes ("the run gain is unlocked by aid-station discipline and heat tolerance, not run fitness") and cannot represent the central bike-pacing-versus-run-cost tradeoff. Recommendation: reconcile repo and VM, re-derive the anchor from 0.636, cap predicted IF at a physiological ceiling (~0.72-0.75 for long course, currently unbounded), and decouple the run estimate from bike IF.

**P1-5. Taper length exceeds both evidence and the project's own spec.** A's generated taper is 4 weeks (26 days); the blueprint's own IM taper is 3 weeks and the meta-analytic optimum is ~2 weeks / 8-14 days. B's taper is 3 weeks against a 2-week spec, and her race-day CTL milestone (84) essentially equals peak (85), implying a taper that sheds almost no fatigue (TSB will not rise). C's 1-week taper is fine. Note this sits behind P0-2: shape the taper first; length is the second-order fix.

**P1-6. Athlete C's targets are internally incoherent and unreachable.** Derived milestones put End-Build CTL 22 -> Peak 29 -> Race-day 40, i.e. race-day CTL above peak, which is impossible under a tapering model. Cause: peak is achievability-capped from a low starting CTL (`ironman-analysis/primitives/load.py:536-571`) while race-day is the raw `race_min`. Compounding, `max_hours_per_week: 4` is enforced nowhere in the planning path, so at a realistic ~196 TSS/week ceiling the race_min is unreachable. A beginner is pointed at a peak he cannot reach on a chart that contradicts itself. Recommendation: reconcile `race_min` against the achievable peak and enforce the hours ceiling at plan time.

**P1-7. Concurrent load for C is invisible to the engine.** `athletes/<C>/current-state.md` records CrossFit 4x/week with the note that this fatigue is not captured in intervals.icu. The planner sizes his cycling purely off ICU CTL/ATL, so it systematically under-counts his true stress: the exact concurrent-interference failure mode, unmanaged.

**P1-8. Threshold integrity: raise-only sync plus stale placeholders.** `sync_ftp_from_eftp` refuses to lower configured FTP when eFTP decays (`lib/thresholds.py:92-130`). The upward logic is right, but in a no-test regime a detraining athlete keeps stale-high zones with no alert for downward drift. A's run threshold (4:02/km) and "running FTP 500 W" are explicit placeholders predating months of no quality running; harmless today (all runs are Z2) but they will silently seed race-pace targets the moment quality resumes. The passive validators that should catch this (`run_durability.py`, decoupling logs) are near-empty (one entry each). Recommendation, within the no-test constraint: add a downward-drift alert (flag, do not auto-cut, when eFTP sits below configured FTP by >X% for N weeks), populate and trend the decoupling/efficiency logs, and add a passive run-threshold estimator from GAP-at-HR on easy runs.

**P1-9. Run-volume caps are bypassable and drift from the stated rule.** The weekly-mileage and long-run caps live only in the stage-1 generation path (`scripts/stage1-plan.py:85-101,159-172`); `validate_week` and `plan_audit` have no run-volume check, so any chat-path push or manual ICU edit skips them. The code cap is the best of the last 4 weeks x 1.15 (`lib/session_library.py:133-134`), i.e. up to +15% off a 4-week peak, while A's own hard rule is "<=10% week-on-week" (`athletes/<A>/reference/rules.md:23`). If the ICU fetch throws, both caps silently become `None` for that week (`lib/session_library.py:135-136`).

**P1-10. The chat layer carries stale and contradictory context for B.** B's `system_prompt.txt` states "Injuries / constraints: None currently" while her live state has an active ankle concern, contains no menstrual content while she is mid-luteal, and hard-codes ramp/CTL numbers (+8%/week, CTL 72-78) that contradict the config (6.0/week, CTL 84-88) and the live `required_tss`. So an in-the-moment "should I do the tempo?" is answered against a prompt that believes she has no injury, no cycle, and different targets than the planner uses.

**P1-11. Fuelling: contradictory bike carb targets across sources.** A's on-bike carb target appears as 60-70 g/hr (`race-plan.md`), 90-100 g/hr (`run-execution.md`), and 90 (`config/athletes.json`, `generate-blueprint.py` 75-90). Reconcile to one figure (likely ~90, consistent with config and modern high-carb consensus). The fuelling model itself is sound; this is doc drift feeding gut-training prescription.

### P2 - polish

- **P2-1.** `recovery_score.compute` (`lib/recovery_score.py`) produces a composite GREEN/AMBER/ORANGE/RED score that only ever becomes card text in the morning check-in; it never modifies a workout. Two "recovery" systems (this and the R-rules) can disagree with no reconciliation.
- **P2-2.** Three-way TSB-threshold incoherence: daily modulation hardcodes -25/-15 uniformly (`modulation.py:35-36`), `tsb_overreach_threshold` in the per-athlete config is read only by `weekly-summary.py`, and `recovery_score` uses a third band set (-15/-25/-35). The per-athlete TSB tuning is nearly cosmetic. Consolidate to one config-driven source.
- **P2-3.** Fatigue on already-easy days is invisible: every modulation rule except heat gates on `_is_quality()` (`modulation.py:187,209,228,250,274`), so a chronic "RPE 8 on easy" signal (present in A's log) produces no response, and only R1 (ankle) can ever prescribe full rest. Add an easy-day fatigue rule.
- **P2-4.** Crash bug: `recovery_score._parse_wellness` returns 4 values on empty input but callers unpack 5 (`recovery_score.py:216` versus `morning-checkin.py:428`), raising a ValueError that is swallowed by a broad except, silently disabling recovery scoring when wellness is empty.
- **P2-5.** Beginner (C) has no `nutrition_target_g_hr`, so the daily nudge defaults to 90 g/hr (`morning-checkin.py:474`), too aggressive for a beginner and contradicting the sportive blueprint's own 60-75. Set an explicit target.
- **P2-6.** Heat: `ACCL_TAU_DAYS = 21.0` is commented "3-week half-life" (`lib/heat.py:61`), but a 21-day time constant is a ~14.6-day half-life. Also `refresh-site-data.py:526` hardcodes a heat "protocol_start_date" of 2026-05-15 that disagrees with the blueprint's formal block start (2026-08-22); confirm these are deliberately different concepts.
- **P2-7.** Heat is off for C despite a late-August alpine sportive whose valleys regularly exceed 30C and a beginner's lower heat tolerance. Make this a deliberate decision rather than a silent default.
- **P2-8.** Tempo counts as scalable "endurance": `_is_endurance` treats any segment below IF 0.85 as flexible volume (`stage1-plan.py:35-46`), so bike tempo at IF 0.80 can be stretched up to 2.2x to close a TSS shortfall - grey-zone inflation. Restrict the TSS-closing lever to true Z1-Z2 (IF <= ~0.75).
- **P2-9.** Strength, swim TSS distort the CTL-target loop: a 40-min strength session books 20 TSS (`planned_tss.py:35`) and swim TSS counts 1:1 with bike/run in the CTL-derived target, so the loop chases a cardiovascular CTL goal with phantom or low-systemic-cost load. Coarse but worth noting.
- **P2-10.** No strength at all for B and C (neither has the `strength_programme` flag). Defensible as a coaching choice; worth a conscious decision.
- **P2-11.** Day-rule validator is inert for B and C: only A has structured `day_rules`, so their real schedule constraints (in prose) cannot be enforced by `validate_week`.
- **P2-12.** Phase for the brief is computed at `today`, not the planned week (`session_library.py:103-106`), while the validator keys distribution to the week start; at every phase boundary the menu and TID come from the outgoing phase while validation uses the incoming one.
- **P2-13.** ICU `menstrualPhase` enum is mapped defensively but unverified against real data (`menstrual.py:26-28`); the chat layer is cycle-blind for B despite tracking being on.

## 4. Keep list - strengths that must survive any rework

- **Deterministic dose gating.** Phase-gated session menu, VO2 unlocked only in late build, progression indexed by weeks-since-unlock with a mandatory intro ramp (`lib/session_library.py:70-79,155-167`). This is the best-engineered part of the intensity system and genuinely prevents a late-unlocked hard type jumping to its hardest dose.
- **Planned TSS never comes from the LLM.** Stage 2 renders load deterministically and explicitly refuses the model's guessed number (`lib/plan_builder.py:82-85`); the single-sourced Banister maths (`load.py:32-33,507-533`) has no dual-implementation drift.
- **Fail-safe push ordering.** New sessions are pushed before old ones are deleted, so a failure can never empty an athlete's week (`plan_builder.py:129-149`).
- **The fuelling model.** Deterministic gap-closing ramp with a gut-training window, surfaced in the daily prescription and computed from real logged intake (`ironman-analysis/primitives/nutrition.py`, `morning-checkin.py:169`).
- **The heat dose model and block timing.** WBGT-style integrated heat-stress dose with strain and humidity terms, sensor over-read correction, and a formal block correctly timed at race minus 4 weeks (`lib/heat.py`, watchdog policing).
- **The cycle-aware model's posture.** Opt-in, individualised cycle length, anchor moved only by a logged period, phase withheld when the anchor is stale, hedged language, placement-only, never cuts load (`lib/menstrual.py`). This is the correct, humble reading of a weak-consensus literature and must not be "strengthened" into prescribing by phase.
- **The run-cap machinery and concurrent-training rules** where they are wired for A: run-specific ceilings independent of total TSS, and textbook strength placement (never before the long ride, >=8h from quality, deloaded into race week).
- **The recovery gate does fire.** Verified real ICU pushes of BLOCKED and MODIFIED sessions for both A and B; this is better than most systems and the behaviour must be preserved when the backstop is made authoritative.
- **Athlete A's chat scaffold and reference corpus.** Quick-pain-log that flags escalation, verify-before-explaining rule, model routing to the heavier model for swap/replan messages, and a genuinely professional race-intelligence corpus (course IF bands, evidence-grounded 2025 post-mortem, pre-wired decision branches). This is the template B and C should be brought up to.
