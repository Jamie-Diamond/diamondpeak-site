## 1. Key decisions

| Question | Decision | Reason |
|---|---|---|
| IcuSync for multi-athlete? | No — direct Intervals.icu API | Avoids second Claude account; scales to N athletes; faster (parallel pre-fetch vs sequential tool calls) |
| Migrate Jamie off IcuSync too? | Yes, same pass | One code path; IcuSync becomes an optional convenience, not a dependency |
| One bot or per-athlete bot? | One bot, route by chat_id | Simpler to operate; one systemd service |
| Claude account(s)? | One (yours) | All athletes share the VM and the claude binary |
| Athlete data isolation | `athletes/{slug}/` folders | Clean, git-friendly, no cross-contamination |

## 3. Intervals.icu direct API (`lib/icu_api.py`)

**Auth:** HTTP Basic — username `API_KEY`, password = athlete's API key.
**Base URL:** `https://intervals.icu/api/v1/athlete/{athlete_id}/`

| Method | Endpoint | Replaces MCP tool |
|---|---|---|
| `get_athlete_profile()` | `GET /athlete/{id}` | `get_athlete_profile` |
| `get_fitness(days)` | `GET /athlete/{id}/fitness?oldest=...` | `get_fitness` |
| `get_wellness(days)` | `GET /athlete/{id}/wellness?oldest=...` | `get_wellness` |
| `get_training_history(days)` | `GET /athlete/{id}/activities?oldest=...` | `get_training_history` |
| `get_events(start, end)` | `GET /athlete/{id}/events?oldest=...` | `get_events` |
| `get_activity_detail(id)` | `GET /athlete/{id}/activities/{id}` | `get_activity_detail` |
| `push_workout(...)` | `POST /athlete/{id}/events` | `push_workout` |

**Key advantage:** all reads can be parallelised in Python with `ThreadPoolExecutor` — the prescription script currently makes 5 sequential MCP calls; with direct API those run in parallel (~1s total vs ~10-15s sequential).

**`fetch_all()` pattern:**
```python
client = IcuClient(athlete_id, api_key)
profile, fitness, wellness, history, events = client.fetch_all(
    "profile", ("fitness", 14), ("wellness", 14), ("history", 14), ("events", 21)
)
# All fetched in parallel — inject into Claude prompt as pre-built context
```

## 4. Bot routing (bot.py changes)

1. On startup: load `config/athletes.json` → build `{chat_id: athlete_slug}` map
2. On message receipt: look up `chat_id` → get slug → load `athletes/{slug}/system_prompt.txt`, `athletes/{slug}/telegram/history.json`
3. Claude call uses athlete-specific prompt + history
4. File writes go to `athletes/{slug}/` paths
5. Unknown `chat_id` → trigger onboarding flow (see §6)

**notify.py** gains `--athlete <slug>` flag to look up the correct `chat_id` from `athletes.json`.

**Cron scripts** gain `--athlete <slug>` flag. The crontab gets one entry per athlete per job:
```
0 6 * * * python3 .../scripts/morning-checkin.py --athlete jamie
0 6 * * * python3 .../scripts/morning-checkin.py --athlete athlete2
```

## 5. Onboarding questionnaire (sport-agnostic)

Delivered via Telegram chat (bot asks one question at a time, stores answers, generates profile at the end). Claude drives the conversation; Python handles the session state.

### Section 1 — Basics
1. Full name?
2. Age and biological sex? (used for training load reference only)
3. Current weight (kg)? And target race weight if you have one?

### Section 2 — Target event
4. What's your A-race? (name, date, location)
5. What distance/format?
   - Running: 10k / Half marathon / Marathon / Ultra (specify km)
   - Triathlon: Sprint / Olympic / 70.3 / Full Ironman
   - Cycling: Gran Fondo / Stage race / TT / other
6. What's your A-goal (time target)? B-goal (acceptable)? C-goal (just finish)?

### Section 3 — Current fitness
7. How many hours per week are you training right now?
8. Recent benchmarks (whatever you have):
   - Run: best 5k, 10k, or half in the last 6 months?
   - Bike: FTP if known, or a recent ride with power data?
   - Swim: CSS or recent 400m time? (tri athletes only)
9. Do you train with power (bike) and/or heart rate?
10. Do you use Intervals.icu? If yes, share your athlete ID and API key (from ICU settings → API).

### Section 4 — History
11. How many years have you been in endurance sport?
12. Longest event you've completed to date?
13. Most recent A-race result (time + rough feel — bonked? strong finish? PB?)?
14. What's your peak training volume been? (hrs/wk or km/wk)

### Section 5 — Constraints
15. Which days can you train? (Mon–Sun, pick any)
16. Maximum hours per week you can commit?
17. Any current injuries or recurring niggles? Where, how long, severity 1–10?
18. Equipment: turbo trainer / treadmill / open water / track / pool?

### Section 6 — Preferences
19. Coaching style: data-heavy (power/pace/zones) or feel-based (RPE + narrative)?
20. Known weaknesses to target in this build?
21. Anything else I should know before we start?

## 6. Athlete generation (`onboarding/generate-athlete.sh`)

After the questionnaire is complete:

1. Claude reads questionnaire answers
2. Selects appropriate rule template:
   - Triathlon (any distance) → `rules-triathlon.md` base
   - Running → `rules-running.md` base
   - Cycling → `rules-cycling.md` base
3. Generates `athletes/{slug}/`:
   - `profile.json` — structured data from answers
   - `system_prompt.txt` — filled from template with athlete specifics
   - `current-state.md` — initial state (injuries noted, open actions seeded)
   - `current-state.json` — machine-readable twin
   - `reference/rules.md` — adapted from sport template + athlete constraints
   - `reference/risk-register.md` — seeded with known risks from answers
   - `reference/decision-points.md` — key milestones to race day
   - Blank logs: `session-log.json`, `swim-log.json`, `decoupling-log.json`, etc.
4. Appends athlete entry to `config/athletes.json` (inactive until admin approves)
5. Notifies admin (you) via Telegram: "New athlete profile ready for review: {name}"
6. **Admin approves** → sets `"active": true` → athlete's cron entries added → first plan generated
